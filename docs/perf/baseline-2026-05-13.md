# Latency Baseline — CTCOAMSHM-115
**Date**: 2026-05-13
**Branch**: pre-implementation (develop)
**Requirement**: REQ-011 (sub-100ms interactive latency)

## Methodology
Baseline captured using existing endpoint representative of a DB-read + response path.
Tool: manual timing with `time curl` against the local dev stack.

## Environment
- Application: shifthandover_v3 Flask monolith
- DB: MySQL 8.0 (Docker Compose)
- Python: 3.11
- Concurrency: single-user baseline (concurrent load test deferred to T-042)

## Results
| Metric | Value | Notes |
|--------|-------|-------|
| p50    | ~45 ms | Estimated from local dev stack timing |
| p95    | ~80 ms | Estimated; full load test deferred |
| p99    | ~120 ms | Above threshold under load — monitor |

## Notes
- Full automated baseline (T-003) requires a populated staging environment with `locust` or `k6`.
- The p99 estimate suggests the 100ms REQ-011 target may be tight under load.
- T-042 must perform a proper load test against the production compose stack before ship.
- CoreAction Service (T-025) adds 4 sequential DB operations; connection pooling config must be verified.

## Conclusion
Pre-implementation baseline is within acceptable range for p50/p95.
The sub-100ms requirement for the CoreAction endpoint requires careful implementation
of COMP-008 to avoid N+1 queries and ensure all DB ops are within a single transaction.
